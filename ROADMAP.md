# NexoraAI Roadmap

The plan for going from **1 to 10**: first make the engine faster on the same hardware, then make small models act like masters, then give the agent eyes, and finally widen the platform. Each item is checked off as it ships.

**Guiding principle** (same one that built this app): don't ask the model nicely — make failure impossible, verify everything deterministically, and hide model mistakes from the user.

---

## Phase 1 — The Engine (v0.10.x)

Speed and context wins that benefit every user, on the exact same models.

- [x] **1.1 Flash attention + KV cache quantization** — enable `flashAttention` and a quantized KV cache in the inference worker; roughly doubles the usable context window in the same RAM. Cheapest win first. *Shipped: the worker now tries flash attention + Q8_0 KV first (context tiers doubled) and falls back to the old F16 path per size on unsupported models/hardware. Verified live: a Qwen2.5-Coder-3B that previously got a 16k window now loads with its full 32k train context on the same 16 GB laptop.*
- [x] **1.2 Partial GPU offload** — a GPU-layer slider plus VRAM-aware auto-recommendation in the Hardware Advisor. Small VRAM cards (e.g. 4 GB RTX 2050) currently fall back to pure CPU; offloading half the layers can nearly double speed. *Shipped: the worker now offloads layers with a step-down ladder (requested/auto → 60% → 40% → 20% of the model's layers → CPU), because Vulkan's free-VRAM estimate is optimistic and even `auto` can OOM while loading weights. Settings gained a GPU-layer slider (0 = auto-fit to free VRAM) and the model card shows the real offload ("GPU 16/29"). Measured on the 4 GB RTX 2050 dev laptop with Qwen2.5-Coder-7B: 4.2 → 7.2 tok/s (+71%), context 16k → 32k.*
- [x] **1.3 Per-phase sampler presets** — plan/enhance = 0.7 temperature (creativity), code generation = 0.2, "düzelt" fix = 0.05–0.1 (surgeon's hand). Three-line change, measurable quality. *Shipped: the single `chat.send` site in the app store now derives sampling from the turn's phase flags — plan/enhance turns get 0.7/topP 0.95, code stays at 0.2, and every fix turn (including the automatic re-tries) runs at 0.1.*
- [x] **1.4 Context compaction with summary** — before the 75% reset, have the model write a 3-sentence summary and carry it into the fresh session, so "what the user accepted/rejected" survives compaction. *Shipped: when the window passes 75%, the worker first asks the still-alive session for a 3–5 sentence summary (never streamed to the UI, capped at 250 tokens, skipped when no headroom remains) and embeds it in the fresh session's compaction note; on any failure it falls back to the old summary-less note. Verified live: a planted fact ("Fırın Luna", purple theme) survived a real 12.5k/16k compaction and was recalled correctly afterwards.*
- [x] **1.5 Migrate main inference to `llama-server`** — the app already ships llama.cpp's official server for vision. Moving chat inference to it unlocks prompt caching / KV reuse (iterations stop re-prefilling the whole project every turn — the single biggest latency cost on CPU), parallel slots (vision + code at once), and speculative decoding (0.5B draft + 7B main, 30–60% faster on code). *Shipped: chat inference now runs through llama-server (auto-downloaded, GPU build preferred — Vulkan on Linux/Windows, Metal on macOS) with flash attention + Q8_0 KV, `--cache-reuse`, `-ngl auto` and a spawn ladder; chat history lives in the main process and shared prefixes hit the server's prompt cache (measured: a 7.7k-token project prefix re-prefilled in 0.14s instead of 23.5s — ~170×). The CJK ban became a per-request logit_bias (token IDs extracted by a 1-second vocab-only scan, cached per model), compaction-with-summary moved up with the history, and the node-llama-cpp worker remains as an automatic fallback engine — verified live by making the server binary unexecutable.*

## Phase 2 — Making Small Models Masters (v0.11.x)

A small model's weakness isn't intelligence, it's freedom. The less freedom, the more mastery.

- [x] **2.1 GBNF grammar-enforced edit blocks** — constrain output at the sampler level so the SEARCH/REPLACE format *cannot* be violated. Retires the streaming watchdog and corrective retries; the same philosophy as the CJK token ban. *Shipped: UPDATE turns now run under a GBNF grammar — SEARCH is hard-capped at 12 lines (verified live: a 3B trying to copy the whole file was cut at exactly line 12 and forced into REPLACE), REPLACE at 40, edit targets are enumerated from the real project paths so a nonexistent file cannot be edited, new-file/[DELETE]/agent-directive shapes stay legal, and pure questions get a bounded `ANSWER:` escape (backtick-free, ≤8 lines) that the chat strips before display. Two grammar holes were found by testing and closed (a `lang="edit"` ambiguity that let the model skip the SEARCH marker, and block-shaped lines leaking into the ANSWER escape); a 59-optional character chain was rewritten after it blew up llama.cpp's grammar-state tracking. A malformed grammar can never brick a turn: if the server rejects the request before any token streams, it is retried without the grammar. The renderer watchdog stays as the safety net for the worker fallback engine. End-to-end in Electron: "rename the heading" produced a single 1-line SEARCH block through the full app path.*
- [x] **2.2 Plan → file-by-file generation** — for big requests: approved plan, then one fresh-context prompt per file ("write only `Hero.tsx`; here are `data.ts`'s types"). Small models are consistent within one file — this generalizes that insight to multi-file projects, with per-file retry. *Shipped: plan turns now emit a grammar-enforced `N. path — description` file list; approving it starts an orchestrated build — files are deterministically reordered (css/lib foundations → ui → sections → entry last) and each is generated in its own short turn, grammar-locked to exactly one fenced block for exactly that path, with lib/css contracts embedded, one retry per missing file, auto-apply per file (the plan approval was the consent; undo stays available) and a final build report. Previous files ride along in the engine history, which the server's prompt cache makes nearly free. Verified live with a 3B: an 8-file cafe-site plan parsed cleanly and per-file turns produced single clean blocks in 6–14s each. The hard-won lesson: a grammar must allow EOS immediately after the closing fence — a mandatory trailing newline masks EOS and sends the model into an `[END_OF_TEXT]` spiral until the token cap; plan-line paths also need parser-side sanity filters (length/depth/text-extension) against in-path repetition spirals. If no file list can be parsed (e.g. worker fallback engine), the old single-shot build runs unchanged.*
- [x] **2.3 Auto-verify after every generation** — run the existing background build check after *every* generation (not just Run), silently auto-fix on failure, then announce "generated *and* compiled". The user never sees broken output. *Shipped as two layers: an instant Babel syntax check of the generated files (milliseconds, no node_modules needed — catches the unterminated-quote/brace/JSX class that dominated the real 14B campaign, verified against broken TSX/TS fixtures with exact line/col output) plus the full vite build check when the project's node_modules already exists (a fresh project never triggers a silent minutes-long npm install — full compile still happens at Run). Failures feed up to two hidden "düzelt" rounds — no user bubble, no technical prompt — announcing only when something was found ("sessizce düzeltiliyor"), fixed ("otomatik giderildi — çıktı doğrulandı") or given up on; clean generations stay silent. Planned builds verify once at the end of the sequence rather than per file, and the loop is guarded against the Run-triggered fix flow, user-initiated turns and IPC settle races.*
- [x] **2.4 Section template bank** — 10–15 hand-crafted, design-proven parametric section templates (Hero, Pricing, FAQ, Footer…). The model fills in content and theme instead of writing from scratch: 3B + good template beats 14B + blank page. *Shipped: 11 premium Tailwind skeletons (hero, navbar, features, gallery/menu grid, pricing, testimonials, FAQ, CTA, contact, footer, about) matched by TR+EN keywords on the planned file's path and description (order-sensitive: hero beats navbar's "header", a priced food menu beats the pricing table) and embedded into that file's planned-build prompt. The decisive lesson came from live testing: a 3B copied natural-sentence placeholders verbatim while happily adapting the color theme — small models follow the example harder than the instruction. Rewriting every placeholder as an explicit `{{MARKER}}` flipped the result completely: brand content in, amber theme in, structure kept, zero markers left, 16s on the 3B. A deterministic retry catches any file still containing unfilled `{{ }}`.*
- [x] **2.5 Model-family prompt profiles** — detect the family (Qwen/Gemma/Llama) from GGUF metadata and apply family-specific quirk fixes, the second axis next to size-adaptive prompting. *Shipped: `detectFamily` classifies qwen/gemma/llama/deepseek/phi/mistral/generic from architecture + name + filename; a short family note (output-discipline / preamble suppression for the chattier families, none for the Qwen baseline) threads through the engines into the system prompt. Testing on a real Gemma surfaced and fixed two bugs: reasoning models returned empty output (fixed by disabling thinking via `chat_template_kwargs`), and — caught by the owner — conversational messages triggered the whole build pipeline (fixed with a `looksLikeBuildRequest` gate so a greeting gets a plain reply while a real build request still enhances/plans). Unit tests 10/10 (family) + 20/20 (build-intent); both fixes GUI-verified end to end.*

## Phase 3 — An Agent With Eyes (v0.12.x)

- [x] **3.1 Import existing projects** — "Open folder" brings any real project into the workspace; smart context and surgical edits already work. Turns the app from a site generator into a local coding assistant for every React project. *Shipped: sidebar "Klasör Aç" imports and LINKS a real folder — a safe scan (build dirs skipped, text-only, capped) loads the files, and a linked-folder registry redirects the whole workspace (edits, Run, dev server, build check) to the original folder, writing only changed files and never scaffolding over a real project. Live-verified: an external Vite+React project was imported and a 3B's surgical edit landed in the original folder's git diff. The test also caught and fixed two real bugs: plan mode hijacking small iteration requests into invented rebuild plans, and the 3.2 runtime hook POSTing to a raw \${RUNTIME_ERROR_PORT} placeholder URL (self-healing had been silently dead).*
- [x] **3.2 Runtime error capture** — inject a tiny error hook into the dev-server page (`window.onerror` + console.error → local endpoint → chat), so "düzelt" covers runtime errors, not just compile errors. *Shipped beyond the original scope: the hook (injected into the project's index.html at sync time, invisible in the code view) posts uncaught errors and rejections to a local collector, and the app starts a hidden fix turn AUTOMATICALLY — zero human input, two attempts per error signature, honest hand-back after that. Stack URLs are rewritten to project paths so smart context targets the right file. Born from the live-test finding that a missing `useState` import compiles cleanly and dies only in the browser; verified end to end with a synthetic error → capture message → automatic fix turn in four seconds.*
- [x] **3.3 Visual self-review** — after Run, screenshot the page and show it to the app's own vision model: empty sections? overflow? does it match the reference image? Auto-iterate if not. The app can already see — it just doesn't look at its own work yet. *Shipped with a deterministic twist: the dev page is captured offscreen and FIRST measured by a pixel scan (dominant-color ratio ≥0.98 = blank page — live testing showed the VL-3B answering 'OK.' to an all-white page, so blankness is measured, not asked), THEN reviewed by the vision model for concrete defects when it's already on disk. Defects start a hidden fix turn feeding the new autonomous repair ladder (rung 0: model-free deterministic fixes for missing imports/apostrophe strings/broken import paths; rung 1: line-numbered context + no-op turns escalate instead of burning retries; rung 3: verify-clean versions are tagged green and the project is auto-restored to the last working version when all else fails). Zero-touch proof: an imported project with a deliberate missing-useState import was detected from the live page and repaired on disk with NO model loaded.*
- [x] **3.4 Git-based generation history** — every accepted generation becomes a commit; unlimited timeline instead of the 20-step undo buffer. *Shipped: post-verify auto-commits each generation (visible request as subject), a Geçmiş tab lists the timeline and restores any version loss-free (safety commit + read-tree + restore commit). git-less systems degrade silently; linked/imported folders are never touched. Live-testing surfaced four real bugs fixed along the way — most notably uncapped requests turning aborted turns into slot-locking ghost generations (the endless 'Düşünüyor…'), enhance instructions poisoning engine history into brief-echo, and briefs mentioning 'mobil' hijacking the project profile to React Native.*

## Phase 5 — The Debug Engine (v1.1.x) — THE NEW DIRECTION

**Mission:** build a first-class debugging engine — the capability that defines tools like Codex, Claude Code and Gemini CLI — that **finds faulty code by scanning**, **sees localhost errors the instant they happen**, **pinpoints the root cause**, and **fixes it both without a model (deterministically) and with one** — proven on simple projects first, then hardened level by level. All future development bends toward this.

**Why we can do this:** the pieces already exist in scattered form — the runtime error collector (3.2), the four-rung repair ladder with model-free rung 0 (3.3), the permanent error corpus (`npm run test:corpus`), pixel + vision self-review (3.3), git green-restore (3.4) and hybrid API routing for the hard fixes (4.1). Phase 5 fuses them into ONE engine with ONE brain, then grows the senses and the surgeon.

- [ ] **5.1 Debug Engine core — one pipeline, one brain.** A single `debugEngine` module that owns the whole loop: **Capture → Diagnose → Locate → Fix → Verify**. Every error — compile, runtime, visual, static-scan — enters the same pipeline, carries the same diagnosis shape (file, line, class, signature, excerpt), and exits through the same fix ladder. One visible event log ("Hata Günlüğü") so the user can watch the engine think: what was caught, where it was located, which rung fixed it, how it was verified. Today's scattered call sites (appStore runtime sub, build-check flow, visual review) become thin feeders into this one engine.
- [x] **5.2 Whole-project static scan — find the bug WITHOUT running.** A "Tara" (Scan) action that walks every project file in milliseconds: Babel parse per file (syntax), project import/export graph (unresolved or case-broken imports, imports of nonexistent exports), undefined identifiers and JSX components, React hooks used without import, leftover `{{MARKER}}`s, package.json sanity. Findings are ranked and fed straight into the 5.1 pipeline — the fixable classes repair themselves, the rest get precise line-numbered model turns. Runs on demand, before Run, and on freshly imported projects. *Shipped: `debugScan` (7 detector families) emits every finding as an autoRepair-compatible diagnosis, so scan findings ride the SAME repair ladder as runtime errors — one pipeline, one brain (`debugEngine`, the 5.1 seed). Three surfaces: a "Tara" button in the workspace header (applies model-free fixes + syncs to disk), a silent pre-Run scan (deterministic classes never reach localhost), and an automatic REPORT-ONLY scan on folder import (never writes to the user's original folder without a click). Suite: `npm run test:scan` — 9/9 (clean project = zero false alarms; 5 deterministic classes repaired AND verified; 3 model-tier classes correctly left). Live GUI proof: a project with a case-broken import + missing useState + undefined info.map was imported → auto-report listed all three untouched on disk → one "Tara" click repaired them into the original folder, no model loaded.*
- [x] **5.3 Fault localization — "the problem is HERE".** Map every stack trace / error signature to exact file + line + symbol (vite URL → project path is already done; extend to source-mapped lines, truncated stacks and minified frames). When the stack doesn't say, RANK suspects: import-graph distance from the crash site, most-recently-edited files (git history is already there), error-class priors from the repair telemetry. The chat answer changes from "something is broken" to "src/components/Menu.tsx:41 — `items` is used but never defined; 92% this file". *Shipped: `faultLocate` — direct-hit mode pulls file+line+SYMBOL from stack frames (92% confidence); truncated-stack mode ranks suspects by undeclared-identifier use (strongest signal), `.prop`-access placement for property-read crashes, and recency tie-break (updatedAt), refusing to invent suspects when there is no signal. Wired into the runtime flow: the 🌐 capture message now carries a "📍 KONUM: file:line (Symbol) — %NN: reasons" block for the user, and the hidden fix turn's line-numbered snippet is re-anchored to the primary suspect — vendor frames' huge line numbers used to scroll the window into the void on truncated stacks. Suite: `npm run test:locate` 6/6; live GUI proof: a prop-less `data.map` crash produced "📍 KONUM: Hero.tsx:18 (List) — %92" in chat. Honest limit: the line comes from vite's transformed module, so it can sit a few lines off the source (the ±12-line snippet window absorbs it) — true source-map decoding remains future work.*
- [x] **5.4 Live localhost watch — see EVERYTHING, instantly.** Today the hook catches `window.onerror` + unhandled rejections. Extend the injected hook to also stream `console.error`, failed network requests (4xx/5xx/refused), React error-boundary catches and vite's HMR error overlay events to the collector — deduped by signature, throttled, each one entering the 5.1 pipeline the second it fires. The dev page becomes a patient on a heart monitor. *Shipped: the hook now has four senses, each tagged with a `kind` so the pipeline picks the right policy — `error` (crashes → the existing auto-repair ladder), `console` (console.error carrying /error/i — React error-boundary reports ride this — same ladder), `network` (fetch ≥400/refused + broken `<img>`/`<script>` resources → an honest 📡 notice with "düzelt" hand-off, deliberately NO automatic model turn: a dead external API is not fixable by editing files), `hmr` (vite's error overlay read from its shadow DOM via MutationObserver → the compile-error protocol; the only signal that exists when a file is broken by hand, since window.onerror never fires). Signature dedup + a 20-report cap per page load; the hook's own collector POSTs are recursion-locked. Live-proven: a broken img produced the 📡 notice, and breaking Hero.tsx on disk with an "external editor" put the overlay's exact compile error into chat within seconds.*
- [x] **5.5 The dual-mode surgeon.** Deterministic first, model second, verification always: rung-0 classes keep growing through the corpus discipline (new class seen live → fixture first → deterministic branch); what rung 0 can't prove goes to surgical line-numbered model turns (hybrid API for the genuinely hard ones); every fix re-verified (layer-1 parse + build check + runtime re-listen) with git green-restore as the never-broken floor. Every decision appended to the repair telemetry so weak classes surface from field data. *Shipped: in "Fixes Only" mode the API is now truly the LAST resort — the first fix attempt for any error always runs locally (rung 0 already closed the free classes), and the turn escalates to the API only when the local model failed on the same signature (2nd runtime attempt / final auto-verify round). An optional ask-first switch in Settings ("API'ye göndermeden önce sor") replaces automatic escalation with a one-time 🤝 hint — typing `düzelt api` / `fix api` IS the consent and forces the escalated turn; with the API off nothing changes at all. Routing decisions land in the repair telemetry (api-turn / api-escalated / api-fallback-local). Matrix suite: `npm run test:routing` 8/8. Live E2E against a mock OpenAI endpoint: `düzelt api` routed the fix turn to the API, its surgical edit block landed in the original folder on disk, telemetry logged api-escalated — and the negative check confirmed a plain `düzelt` starts LOCAL generation with zero API requests.*
- [x] **5.6 Simple-project proving ground — the engine earns trust before it gets power.** Grow the error corpus from single-file fixtures into a suite of small intentionally-broken PROJECTS (missing import, dead data var, broken import path, syntax cut, runtime-only crash, blank render…). A harness runs the whole engine against each: measure find-rate, fix-rate, time-to-green. A capability is "shipped" only when the suite says so — same discipline that built Phases 1–4, now automated end to end. *Shipped: `npm run test:ground` runs 10 whole-project fixtures through the REAL pipeline via both entry channels — 'scan' (static discovery) and 'runtime' (verbatim collector diagnoses through locateFault + rung 0) — and prints the scorecard: find-rate 9/9, deterministic repair 7/7 (applied AND verified green), avg time-to-green 22ms (worst 96ms), zero false alarms on the clean project, model-tier classes correctly refused. `npm run test:engine` chains all five engine suites (corpus + scan + locate + routing + ground = 44 checks). The discipline paid out on the FIRST run: P8 (prop-less `.map` crash, truncated stack) exposed a real localization weakness — two files both using `.map` tied and the wrong one won; the engine now ranks a receiver that is an incoming prop/parameter (can be undefined) above one that is concretely declared or imported, and the fix landed with its own regression fixture in the locate suite.*
- [x] **5.7 Advanced debugging — from reactive to proactive.** Multi-error sessions (fix in dependency order, not arrival order), cross-file root causes (the crash is in A, the bug is in B), watch mode (continuous background scan while the user edits, findings appear as they type), and value probes — the engine injects temporary logging into suspect lines, reads the values from the collector, and reasons from DATA instead of guessing. This is where the tool stops imitating Codex/Claude/Gemini and starts being one. *Shipped, all four: (1) multi-error sessions — findings repair in DEPENDENCY order (syntax → imports → identifiers) with multi-pass rescanning, because a syntax break masks the same file's other bugs (P11: apostrophe cut + hidden missing import, both fixed in ONE call); (2) cross-file root cause — on a property crash whose receiver is a prop, the engine names the file that renders the component WITHOUT the prop as "KÖK NEDEN ADAYI", the file the stack never shows; (3) watch mode — an eye toggle in the workspace runs a debounced report-only scan as files change (never edits under the cursor), findings as a badge with file:line tooltips — live it even caught a junk file a model turn had created; (4) value probes — the suspect expression is temporarily wrapped (`data.map` → `__nxProbe('data', data).map`), the page is force-rendered offscreen (React unmounts a crashed tree — HMR alone never re-runs the module), the value at the moment of the crash streams back (🔬 "data = undefined"), the file is restored under guarantee, and the fix turn's prompt starts from MEASUREMENT, not guess. Two hard-won race lessons: the probe waiter must be armed BEFORE the sync (vite's build window ate the data), and probe delivery must bypass ALL busy guards — the same crash emits error+console twice, the second starts a turn, and a busy guard was dropping the measurement (probe-timeout ×4 before the live proof: probe-hit + restored file + 🔬 in chat). Suites: `npm run test:engine` = 6 suites, 51 checks, all green.*

## Phase 6 — The Engine, 10× (v1.2.x) — FROM DEBUGGER TO COLLEAGUE

**Mission:** take the Phase 5 engine from "it catches and fixes" to **Codex / Claude Code / Antigravity class**: compiler-grade eyes, a *real* runtime debugger instead of string probes, edits that cannot miss, verification that *uses* the product like a human tester, and an engine that **learns from its own field data**. Phase 5 built the reflexes; Phase 6 builds the judgment.

**Why these seven (the honest audit):** field telemetry says rung 0 misses **57%** of live errors (39/69 `kat0-miss`); value probes hit only 3/11 in the field and mutate user files to measure; the 14B fabricated SEARCH blocks three times in one night (data-array copy — the matcher has no line/AST anchor to save it); an aborted turn left a 254-div spiral in the store (no turn atomicity); the scanner is regex-grade, so every false-positive class costs a manual fix; "verified" still means *compiles + not blank*, not *works*; and the engine writes telemetry it never reads. Each item below kills one of these, with the industry's current best practice as the bar (Claude Code shipped native LSP diagnostics in late 2025; Antigravity verifies by driving a real browser and hands the user artifacts; Google's agentic-APR work shows reproduction-first repair beats guess-first repair).

- [x] **6.1 A real runtime debugger — CDP `Debugger` domain, not string probes.** Attach Electron's `webContents.debugger` to the offscreen page, `setPauseOnExceptions`, and when the crash fires read the ACTUAL call frames and scope variables at the moment of death — exact file/line from the source map, every local's value, the full async stack — with **zero file mutation** (retires the inject-sync-restore probe dance and its race conditions; keeps `__nxProbe` only as a fallback for non-exception measurements). The fix turn's prompt starts with "at List (Hero.tsx:4), `data` = undefined, caller `<List />` passed no props" — measured, sourced, undeniable. Also closes the 5.3 source-map honesty gap for free. *Shipped: `debugInspect` opens the dev page offscreen with the debugger attached and hands back user-code frames with real locals and the source-mapped ORIGINAL line — live proof: "🔎 Debugger (çökme anı): Hero.tsx:5 — List() · data = undefined", the exact source line of the `.map`, file md5 untouched, probe fallback never needed. The source-map line decoder is dependency-free VLQ (self-validated against esbuild's real maps, `npm run test:sourcemap` 6/6, in test:engine = 7 suites / 57 checks). Three live-diagnosed lessons are IN the code: an absolute outer `Promise.race` because an inner await hung before the inner timer could guard it (and froze the runtime handler's `probing` latch with it); debugger commands never resolve on a webContents that has not committed a navigation — commit `about:blank` first, then enable, then navigate; and pause on `'all'` + skip-vendor-frames-until-user-code, because React catches render errors internally and rethrows from `commitRootImpl` — the 'uncaught' pause fires where the user frame is already gone.*
- [x] **6.2 Compiler-grade diagnostics — TypeScript language service over the in-memory project.** Embed TS's `createLanguageService` on the artifacts store's virtual file map (no node_modules, no install, fully offline): real type errors, truly undefined symbols, unused/missing imports, wrong prop types — the diagnostics class Claude Code gets from LSP, computed on OUR file map in milliseconds. The regex scanner stays as the fast first pass and the fallback for non-TS files; every scanner false-positive class (the `function List({ data })` lesson) becomes structurally impossible. Findings keep flowing into the same 5.1 pipeline diagnosis shape. *Shipped: `tsDiagnostics` runs a cached, incremental language service over the in-memory map — TS's own lib.d.ts set (100 files) and @types/react(-dom) are baked into the bundle as lazy raw chunks at BUILD time (`import.meta.glob` — which silently matches nothing if you forget the renderer's vite root is `src/`, hence the missing-default-lib self-disable guard), an ambient `declare module '*'` keeps unknown bare imports quiet, and a curated code whitelist (2304/2305/2339/2551/2552/2554/2741) keeps the signal clean. ## Phase 8 — Real-Time Hardening (v1.4.x) — THE CLOCK IS THE ENEMY

**Mission:** Phase 7 shipped the agent experience and every browser-harness test was green — then the very first end-to-end run on the REAL app with a REAL model (the Zeytin Kafe live test, 2026-07-07) exposed a whole class the harness cannot see by design: **real-duration races and a busy inference server**. Turns that take minutes, verify chains that overlap timers, an abort that doesn't reach llama-server. The packaging was never the suspect — the identical code stalls under a real clock and flows under a mock one (the same queue that froze in production ran itself in the browser). Phase 8 makes the agent layer honest under real time, and makes the test surface able to SEE time.

**The live-test ledger (why these seven):** the queue processor never self-started (twice) because its only trigger — the `sending` falling edge +1.2s — raced the minutes-long verify/repair chains; the behavior walk's one-shot 12s timer landed on a running auto-repair turn and silently never ran; a watchdog-cut penalty turn hung at 0 bytes for 36 minutes while llama-server chewed a request abort never cancelled — and Durdur made it WORSE (the reality-feedback retry opened a fresh hidden turn), leaving restart as the only exit; a task that never added the requested line was stamped "verified" (the verdict measures compile-clean, not goal-met); and everything the engine learned went into the default `nexora-projesi` folder because planned builds don't write a package.json.

- [x] **8.1 The hang chain — no turn may ever freeze the app.** A stream-liveness sentinel (N seconds with zero bytes on an active turn = the turn is declared dead, honestly reported, engine unlocked); Durdur becomes ABSOLUTE — it also cancels the hidden retry/penalty/queued-hidden turns instead of letting one reopen the wound; and abort must actually cancel the llama-server task (server-side cancellation / connection teardown that stops decode, verified against a deliberately busy server). The 36-minute zombie becomes structurally impossible.
- [x] **8.2 The queue heartbeat — delegated work may never sleep.** Replace the single-shot falling-edge trigger with a persistent heartbeat: while anything is queued, the processor re-knocks until the engine is genuinely idle, and the inbox card says WHY it waits ("motor meşgul: onarım turu koşuyor") instead of a frozen "sırada". Verified against real multi-minute verify chains, not mock timing.
- [x] **8.3 Behavior walk that retries.** The one-shot 12s timer becomes schedule-until-done: if the engine is busy when it fires, it re-arms (bounded, logged) — after Run, the walk ALWAYS eventually runs or honestly reports why it could not.
- [x] **8.4 Goal-met verdicts — "verified" must mean the request happened.** A task's verdict gains a goal check: deterministic content assertions derived from the prompt where possible (requested text/values present in touched files), behavior-walk evidence where visible — a task that compiled but didn't do the job lands as "⚠ incele: istek karşılanmadı", never "verified".
- [ ] **8.5 Project identity — no more default-folder pileup.** Planned builds write a real package.json (name derived from the brief), the identity fallback warns loudly, and knowledge/rules/history stop silently commingling under `nexora-projesi`.
- [ ] **8.6 Small honest fixes from the ledger.** The plan derives a Gallery section when the brief asks for one; task turns re-arm the walkthrough context so the document keeps versioning after queue work; inbox timestamps/status text stay live.
- [ ] **8.7 A test surface that can see time.** The browser harness gains a timing-realistic mode (multi-second turns, slow verify chains, a busy-server simulator) so race-class bugs die in preview; plus a real-engine smoke suite (`test:e2e` with a small real model or a latency-faithful llama-server stub) covering exactly the killers: queue-under-load, abort-against-busy-server, behavior-timer-vs-repair-turn.
